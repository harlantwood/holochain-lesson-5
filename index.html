Capability Tokens & P2P

Zome Calls

  • Zome calls have a unified security model, that protects all zome calls independently of where they originated
  • When the conductor receives a zome call request, it checks whether the zome call complies with the security checks:
    • If the call is signed by the agent with the public key for the cell, the call is accepted and executed
    • If not, the conductor looks up the source chain for that cell for capability tokens for the requested function
      • If some capability token allows the request, it is executed
      • Otherwise, the request is rejected

    • Capability Tokens (I)

  • Private entries called "Capability Grants" that grant authorization to execute zome calls by sources external to the cell
  • Can be created as any other entry, at any point in time
  • When they are deleted, that grant is not longer taken into account
  • This means that the cell is always in control of the grants that it makes to external sources

    • Capability Tokens (II)

    Types

  • Transferable: any source that knows a specific secret
    • The zome call must carry a secret
    • Only the calls that carry the secret specified in the capability token will be executed
    • The secret can be stored in the source chain of the caller with a "Capability Claim" entry
      • Counterparty to "Capability Grant"
      • Private entry, not used in the security checks
      • Only for convenience
  • Assigned to specific agents
    • The zome call must carry the agent public key of the caller and its signature and a capbility secret
    • Only the calls that carry the secret specified and are made by one of the agents specified in the capability token can make the call
  • Unrestricted: disable security checks
    • Any call to the function coming from any source will be executed

    • Capability Tokens (III)

#[hdk_extern]
fn allow_agent_to_call_zome_function_a(authorized_agent: AgentPubkey) -> ExternResult<()> {

  let zome_name = zome_info()?.name;
  let function_name = FunctionName(String::from("zome_function_a");

  let mut functions: BTreeSet<(ZomeName, FunctionName)> = BTreeSet::new();
  functions.insert((zome_name, function_name));

  let mut assignees: BTreeSet<AgentPubkey> = BTreeSet::new();
  assignees.insert(authorized_agent);

  let access = CapAccess::Assigned {
    secret: create_password(),
    assignees
  };

  let capability_grant = CapGrantEntry {    
    tag: String::from("zome_func"),
    access,
    functions
  };

  create_cap_grant(capability_grant)?;

  Ok(())
}

    Paths (I)

  • Anchors are great, but they can introduce problems
    • If a large amount of links are attached to the same base, they can become a burden of storage and bandwidth for the cells holding those links
      • This is know as a "DHT Hotspot"
    • They don't work well with complex queries that need filtering, sorting, etc.
      • For example, a query of the form "get_comment_by_date(start_date: Timestamp, end_date: Timestamp)"
  • Solution: paths

    • Paths (II)

  • Built-in to the HDK
  • Paths are anchor trees
    • Top level anchor acts as the root node
    • This root node is linked to its children, which themselves are anchors
    • This tree can be dynamically constructed and have arbitrary depth
  • They are composed of a vector of components
    • Each component is an arbitrary byte array
    • Normally constructed from a string

    • Paths (III)
